5 eMASS Mistakes That Delay Your ATO by Months

These are the documentation errors and artifact gaps we see repeatedly on RMF packages — and every one of them is preventable if you know what Authorizing Officials actually care about.

After managing eMASS packages on multiple DoD programs — including a recent 7-month ATO compression on AWS GovCloud at IL5 — I've seen the same mistakes cause the same delays across nearly every authorization effort I've encountered. These aren't obscure edge cases. They are fundamental documentation errors that add weeks or months to your timeline because each one triggers a review cycle that could have been avoided entirely.

Mistake #1: Phantom Section References in Security Control Narratives

This is the single most common artifact error I encounter. A security control implementation narrative will reference a section of the System Security Plan, a specific figure, or another document — and that section doesn't exist, has a different numbering scheme, or was removed in a previous revision. When an SCA reviewer flags a phantom reference, your entire narrative for that control family becomes suspect, and they start looking more closely at everything else.

The fix is straightforward but requires discipline: every time you update your SSP, IRP, or ISCP, you must audit all cross-references in your eMASS control narratives. Use a consistent section numbering scheme from day one and maintain a reference index. This is tedious work, but it prevents the most common review rejection I've seen.

Mistake #2: Version Label Mismatches Between Documents and eMASS

Your eMASS artifact record says "IRP v2.0 Final" but the document header says "IRP v2.1 Draft." Your SSP in eMASS is labeled "Rev 3" but the file metadata shows it was last modified two months after the upload date. These mismatches signal to the reviewer that your configuration management process is broken — and if you can't manage document versions, they reasonably question whether you can manage system configurations.

Implement a strict document control process: version numbers in the filename, the document header, and the eMASS artifact description must match exactly. When you upload a new version, update all three simultaneously. Log every version change in a document revision history table at the front of the artifact.

Mistake #3: Incomplete Figure Narratives in the SSP

System Security Plans routinely include network diagrams, data flow diagrams, and architecture figures — but the narrative text around those figures often fails to describe what the figure actually shows. A reviewer sees "Figure 4-2: Network Architecture" followed by a diagram, followed by text that doesn't reference the diagram at all. This creates an evidence gap: the figure may show the right architecture, but without narrative context, it doesn't demonstrate that the author understands what's depicted or why it matters for the control being addressed.

Every figure in your SSP should have a preceding introductory sentence explaining what the figure demonstrates, and a following paragraph that describes the security-relevant elements shown. This is not optional decoration — it's the connective tissue between your architecture and your control implementation claims.

Mistake #4: Confusing Implementation Completeness with Documentation Completeness

This is the most subtle mistake and the most expensive. A security control can be fully implemented in your environment — MFA is deployed, audit logs are configured, access controls are enforced — but if the documentation doesn't describe that implementation with specificity, evidence references, and responsible parties, the control will be assessed as "partially implemented" at best.

The inverse is also true and equally dangerous: beautiful documentation that describes a control implementation that doesn't actually exist in the environment. Assessors verify. They check configurations. They interview operators. If your narrative describes a SIEM integration that hasn't been deployed, you've created a credibility problem that extends far beyond that single control.

Every control narrative should answer three questions: What is implemented? How is it implemented (specific technology, configuration, procedure)? Where is the evidence (screenshot, export, policy document, with date)?

Mistake #5: Treating POA&Ms as a Dumping Ground

A Plan of Action and Milestones is not a parking lot for things you don't want to deal with. Too many RMF packages contain POA&Ms with dozens of items, unrealistic completion dates, no assigned owners, and no evidence of progress. An AO reviewing a package with 40+ open POA&M items and no demonstrated remediation activity will not sign. Full stop.

A well-managed POA&M has three characteristics: each item has a specific, named owner (not a team, a person); each item has a realistic completion date based on actual resource availability; and each item shows progress evidence at every review cycle. If a POA&M item has been open for 6 months with no status update, it communicates that nobody is managing the risk — and that's exactly the message that delays your ATO.

The Bottom Line

Every one of these mistakes is preventable with proper artifact management and cross-reference discipline. The pattern is consistent: ATO delays are rarely caused by genuine security deficiencies. They are caused by documentation gaps that make it impossible for the AO to confidently assess the security posture of the system.
